# Open Ticket Workspace

Create a per-ticket workspace with Jira details, git repo clone, and tracking files.

## Use When
- User says "티켓 열기", "작업 시작", "open ticket", "open workspace"
- User provides a Jira ticket ID (e.g., PROJ-123)

## Do Not Use When
- User wants to manage Jira tickets without a local workspace -- use `jira-master`

## Steps

1. **Read workspace root**

   Parse `~/.claude/CLAUDE.md` for the `tw-workspace:` line inside `<!--ticket-workspace:start-->` block.
   If not found, tell user to run `/ticket-workspace:tw-setup` first.

2. **Fetch Jira ticket**

   - Call `ToolSearch("+atlassian jira")` to discover MCP tools
   - Fetch ticket with `mcp__mcp-atlassian__jira_get_issue` using the ticket ID
   - Extract: key, summary, type, status, priority, assignee, sprint, description

3. **Check existing workspace**

   Check if `{workspace-root}/{TICKET-KEY}/` already exists.
   - If exists: ask via `AskUserQuestion` -- Resume / Recreate / Cancel
   - If not: proceed

4. **Create workspace structure**

   ```bash
   mkdir -p {workspace-root}/{TICKET-KEY}/{notes,repo}
   ```

5. **Generate workspace files**

   Create these files (use Write tool):

   **`CLAUDE.md`** (Claude Code가 워크스페이스 진입 시 자동 인식):
   ```markdown
   # {TICKET-KEY}: {summary}

   ## Ticket Context
   - **Key**: {TICKET-KEY}
   - **Type**: {type}
   - **Status**: {status}
   - **Jira**: https://{domain}/browse/{TICKET-KEY}

   ## Workspace Files
   - `work-log.md` -- 작업 일지 (날짜별 기록)
   - `status.md` -- 진행 상태 체크리스트
   - `repo/` -- git 저장소

   ## Available Skills
   - `/ticket-workspace:tw-log add "내용"` -- 작업 로그 추가
   - `/ticket-workspace:tw-status sync` -- 상태 동기화
   - `/ticket-workspace:tw-transition` -- Jira 상태 전환
   - `/ticket-workspace:tw-close` -- 워크스페이스 닫기
   ```

   **`.tw-config.json`**:
   ```json
   {
     "ticket": "PROJ-123",
     "summary": "Ticket summary from Jira",
     "type": "Story",
     "status": "To Do",
     "repo": "",
     "branch": "",
     "baseBranch": "",
     "createdAt": "2026-03-18T14:30:00+09:00",
     "workspaceStatus": "active"
   }
   ```

   **`README.md`**:
   ```markdown
   # PROJ-123: {summary}

   | Field    | Value                                           |
   |----------|-------------------------------------------------|
   | Key      | PROJ-123                                        |
   | Type     | {type}                                          |
   | Status   | {status}                                        |
   | Priority | {priority}                                      |
   | Assignee | {assignee}                                      |
   | Sprint   | {sprint}                                        |
   | URL      | [PROJ-123](https://{domain}/browse/PROJ-123)    |

   ## Description

   {description from Jira}

   ---
   > Auto-generated by ticket-workspace. Last synced: {timestamp}
   ```

   **`work-log.md`**:
   ```markdown
   # Work Log: PROJ-123

   > {summary}

   ---

   ## {today's date}

   ### {HH:mm} - Workspace created
   - Workspace initialized
   - Repository cloned (if applicable)
   - Branch: {branch-name} (if applicable)
   ```

   **`status.md`**:
   ```markdown
   # Status: PROJ-123

   > {summary}

   ## Current State

   | Item          | Status    |
   |---------------|-----------|
   | Jira Status   | {status}  |
   | Branch        | --        |
   | PR            | --        |
   | Last Activity | {now}     |

   ## Progress

   - [x] Workspace created
   - [ ] Repository cloned
   - [ ] Branch created
   - [ ] Implementation
   - [ ] Tests
   - [ ] Code review
   - [ ] Deployed

   ## Blockers

   (none)

   ## Links

   - Jira: [PROJ-123](https://{domain}/browse/PROJ-123)

   ---
   > Status markers: [ ] pending, [x] completed, [>] in progress, [!] blocked
   ```

6. **Git repo clone** (if repo URL provided or asked)

   - Ask user for git repo URL via `AskUserQuestion` if not provided as `--repo` argument
   - Option to skip cloning (workspace without repo is valid)
   - If provided:
     ```bash
     git clone {repo-url} {workspace-root}/{TICKET-KEY}/repo/{repo-name}
     ```

7. **Create feature branch** (if repo was cloned)

   - Ask user for base branch (suggest `develop`, `main`)
   - Generate branch name: `feature/{TICKET-KEY}-{kebab-summary}` (same convention as air-claudecode git-branch)
   - ```bash
     cd {repo-dir} && git fetch origin && git checkout -b {branch-name} --no-track origin/{base-branch}
     ```
   - Update `.tw-config.json` with repo, branch, baseBranch
   - Update `status.md` with branch info

8. **Optionally transition Jira** via `AskUserQuestion`

   Ask: "Jira 상태를 'In Progress'로 변경할까요?"
   - Yes: fetch transitions, find "In Progress", execute transition
   - No: skip

9. **Show summary** via `AskUserQuestion` with markdown preview

   ```
   Workspace Created: PROJ-123
   ===========================

   Path:     ~/Desktop/ticket-workspace/PROJ-123
   Ticket:   PROJ-123 - {summary}
   Status:   {jira-status}
   Branch:   feature/PROJ-123-{kebab-summary}
   Repo:     {repo-dir}

   Next steps:
   - cd {repo-dir}
   - /ticket-workspace:tw-log add "started investigation"
   - /ticket-workspace:tw-status update
   ```

## Examples

**Good:**
User: "PROJ-123 워크스페이스 열어줘 --repo git@github.com:org/api.git"
Result: Full workspace with cloned repo, feature branch, all tracking files.

**Good (minimal):**
User: "PROJ-456 열기"
Result: Workspace with Jira info, no repo clone (user skipped).

**Bad:**
User: "워크스페이스 열어"
Action: Create workspace without asking for ticket ID.
Why bad: No ticket ID, no Jira context.
